PHASE 1 — Backend: Fastify + Prisma + SQLite
================================================================================
- [x] Initialize backend: TypeScript, Fastify, `@fastify/cors`, `@fastify/jwt` (or cookie session)
- [x] Add Prisma with SQLite datasource and `prisma/schema.prisma`
- [x] Define models:
      - User (id, username, role: ADMIN | USER, passwordHash)
      - Patient (id, lastName, firstName, dateOfBirth, mrn, cellPhone, createdAt)
      - ProcedureTemplate (id, name, createdByUserId, guestReadOnly flag if needed)
      - TemplateMessage (id, templateId, sortOrder, body, weeks, days, hours, minutes, seconds)
      - ActiveCampaign (id, patientId, templateId, status, startedAt)
      - ScheduledMessage (id, campaignId, templateMessageId, sendAt, sentAt, status)
      - SimulatedSmsLog (id, direction, from, to, body, patientId, campaignId, replyToMessageId)
      - PhysicianForward (id, physicianPhone, enabled) — config for forward targets
- [x] Seed database: admin/user accounts with default password `password` (bcrypt hashed)
- [x] Auth routes: POST `/auth/login`, POST `/auth/logout`, GET `/auth/me`
- [x] Role middleware: admin full access; user standard; guest read-only templates (per todo lines 4–6)
- [x] Patient CRUD + search: GET `/patients?query=`, POST `/patients`, GET `/patients/:id`
- [x] Template CRUD: GET/POST/PUT `/templates`, GET `/templates/:id` with nested messages
- [x] Campaign start: POST `/campaigns` (patientId + templateId) → create scheduled messages from offsets
- [x] Scheduler service: interval/cron job that marks due messages and emits to SMS simulator API
- [x] Inbound simulated reply: POST `/sms/simulate/reply` → log + build physician forward payload
- [x] Physician forward: POST `/sms/forward` or append to right-panel feed via GET `/sms/physician-inbox`
- [x] Health check: GET `/health`
